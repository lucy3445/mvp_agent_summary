Part 2. MVP와 PMF (Product Market Fit)

1. MVP (Minimal Viable Product, 최소 기능 제품)

고객에게 가치를 제공할 수 있는 최소한의 핵심 기능만 구현한 제품입니다. 완벽한 완제품을 개발하여 출시하기 전에, 시장에 빠르게 내놓아 고객의 진짜 반응을 확인하고 비즈니스 가설을 검증하기 위한 필수 도구입니다. 불필요한 비용과 장기적인 시간 소모를 획기적으로 줄여줍니다.

2. PMF (Product Market Fit, 제품-시장 적합성)

우리의 제품이나 서비스가 실제 시장의 강력한 수요를 충족하고 있는 상태를 뜻합니다. PMF는 단번에 완성되는 선형적 도달이 아닌, 아래와 같이 반복적인 순환 루프(Feedback Loop)를 구동함으로써 달성됩니다.

3. PMF 피드백 루프와 비즈니스 성장 전략의 연결고리

그로스 해킹과 피벗은 PMF 피드백 루프와 분리된 개별 개념이 아니라, 루프를 실제로 성공적으로 굴리기 위해 조화롭게 맞닿은 구체적 비즈니스 실행 방식입니다.

그로스 해킹 (Growth Hacking - 루프의 가속화)

빠르게 제품을 성장시키고 사용자를 유치하는 전략으로, 그 작동 원리가 MVP 피드백 루프를 끊임없이 반복 구동하면서 PMF를 찾아 나가는 가속 프로세스와 일치합니다. 시장 데이터 수집 및 분석을 통해 피드백 루프의 속도를 극대화하는 촉매제입니다.

피벗 (Pivot, 사업 방향 전환 - 루프의 궤도 수정)

루프를 도는 도중 수집된 고객의 피드백 분석 결과가 당초 가설과 완전히 불일치할 때, 기존 가설을 고집하지 않고 제품의 핵심 축 및 비즈니스 방향성을 과감히 변경해 새로운 루프를 구동하는 의사결정 방식입니다.

💡 Feedback과 피벗(Pivot)의 성공 사례: YouTube의 기원
유튜브(YouTube)는 본래 "동영상 소개 기반의 데이팅 서비스"였습니다. 그러나 사용자들이 자신의 얼굴이 드러나는 데이팅 소개 영상을 찍어 올리는 것을 강력히 꺼리자 가설 실패를 마주하게 되었습니다. 이때 창업자들은 데이팅 컨셉을 버리고, "주제 제한 없이 누구나 아무 동영상이나 자유롭게 올리고 공유할 수 있는 서비스"로 피벗을 단행했습니다. 시장 피드백을 수용해 방향성을 완전히 수정하여 PMF를 찾아낸 결과, 오늘날 세계 1위 동영상 공유 플랫폼이 되었습니다.

Part 3. 시장조사를 위한 8가지 질문과 시장 규모 측정

1. 기획 단계의 8가지 질문 리스트

비즈니스 설계 시 스스로에게 아래 8가지 질문을 던져야 하며, 답변이 하나라도 막힌다면 첫 번째 질문부터 다시 기획을 시작해야 합니다.

  1. 누구를 위한 서비스인가? (명확한 타겟 고객)
  2. 고객이 유의미한 규모의 집단인가? (시장성 검증 - 💡 Gemini Canvas 활용)
  3. 그 사람들의 어떤 문제를 해결해 주는가? (문제의 명확성)
  4. 정말로 그게 문제가 맞나? (진짜 고통 검증)
  5. 문제를 어떻게 해결할 것인가? (실효성 있는 해결책)
  6. 그 해결 수단이 어떤 특별한 가치를 갖나? (독점적 차별 가치)
  7. 해결 수단을 어떻게 고객에게 전달할 것인가? (마케팅 채널)
  8. 경쟁자 대비 어떤 장점이 있나? (경쟁 우위 분석)

2. 시장 규모의 3단계 레이어 (SOM, SAM, TAM)

시장 규모를 체계적으로 도출하기 위해 3단계 레이어 구조를 갖추어 분석합니다.

  • TAM (Total Available Market, 전체 시장): 제품이 속해 있는 산업의 전체 규모 (예: 전체 자동차 산업 시장)
  • SAM (Serviceable Available Market, 유효 시장): 우리 비즈니스 모델이 도달 가능한 현실적 범위 (예: 국내외 전기차 시장)
  • SOM (Serviceable Obtainable Market, 수익 시장): 우리의 자원으로 즉시 확보 가능한 실질 영역 (예: 국내 전기 SUV 중 4천만원 이하 섹터)
  • Niche Market (틈새시장): 기존 대형 플레이어들이 점유하지 못한 SOM의 좁고 구체적인 틈새 영역입니다.
⚠️ MVP 제작 시 시장 규모 한계선 설정
새롭게 MVP를 구상하고 배포할 때에는 거창한 TAM이나 SAM 규모를 논하기보다는, 틈새시장(Niche Market)과 즉각적인 확보가 가능한 현실적인 SOM 영역까지만 최대 한계선으로 설정하여 목표를 잡아야 검증이 원활합니다.

3. AI 도구 선택 가이드 (강사 설명 요약)

Gemini Canvas

2번 질문 (시장 규모 및 타겟층) 분석 시 최적

Gemini Canvas는 강력한 최신 정보 실시간 탐색 능력을 갖추고 있으며, Canvas 우측 상단의 '만들기 ➡️ 인포그래픽'과 같이 시각화가 가미된 구조적 보고서/표 형태로 즉각 정리해 주는 기능이 매우 우수하므로 2번 질문에 최적입니다.

ChatGPT / Claude

그 외 모든 질문 (1, 3~8번) 분석 시 최적

제미나이는 최근 코딩 및 특정 환경에 다소 깊게 치우쳐 있는 경향이 있어, 유연한 생각의 조율이나 비즈니스 아이디어 구체화, 추상적인 논리 전개 등은 ChatGPT가 인간의 생각과 가장 가까운 맥락으로 문맥을 풍성하고 정교하게 정리해 줍니다.

4. 📝 라이브 질문 실습 프롬프트 전개 과정

실습 과정에서는 디스코드에 공유된 '8개 질문이 적힌 강의안 슬라이드 캡처 사진'을 복사하여 각 AI 채팅창에 이미지로 첨부하고 대화를 이어 나갔습니다.

[2번 질문 실습 프롬프트] (Gemini Canvas에 질문 슬라이드 사진을 붙여넣고 입력)

2번 질문에 대한 답이 필요해. 시장 조사를 해 줘. 상세하게.

[8번 질문 실습 프롬프트] (ChatGPT에 질문 슬라이드 사진을 붙여넣고 입력 - 순차 전개)

• 1차 입력 (질문 전개 및 문제 발생)

8번 질문에 대한 답변이 잘 안 떠오른다. 니가 생각해 봐.
💡 실습 중 문제 상황 및 우회 지침 (상황 설명)
8번 질문은 경쟁사 대비 강점을 묻는 질문인데, 실습 특성상 1~7번 기획안(가짜 배달 앱의 타겟 고객 및 구체적 해결안)이 ChatGPT에 사전 학습되지 않은 상태여서 AI가 제대로 된 경쟁사를 매칭하지 못하고 막혔습니다. 이에 강사(반병현 대표)는 "실습 진행을 위해 1~7번 기획이 성공적으로 잘 완료되었다고 가상의 전제를 깔아준 뒤, 유사 경쟁사(예: 진짜 배달의민족 앱 등)를 임의로 탐색하여 8번 답변을 만들어라"고 우회 지시하는 2차 프롬프트를 전개했습니다.

• 2차 입력 (앞선 기획 완성 가정 하에 경쟁사 탐색 및 답변 생성 유도)

실습용이니까 앞부분이 잘 됐다 가정하고 경쟁사 찾아와서 8번 답변 만들어라.

• 3차 입력 (도출된 정보를 바탕으로 최종 개발용 시방서(사양서) 작성을 지시)

이걸 달성하기 위한 MVP를 만들거야. AI한테 제작 시킬거니까 개발을 위한 시방서를 니가 만들어 줘. 1. HTML, CSS, JS만 사용할 것 (빌드 프레임워크 X) 2. 저작권 문제 없을 것 3. 모바일에서도 잘 돌아갈 것

1. AI 에이전트 입력용 최종 프롬프트 (가짜 배달 웹 앱 개발용)

앞서 ChatGPT가 만들어 낸 개발 시방서(사양서) 텍스트 전문을 복사하여 Antigravity IDE 에이전트 대화창에 붙여넣고, 그 아래에 덧붙여 입력한 핵심 프롬프트 전문입니다.

위 요구사항을 만족하는 웹 앱 만들어 주는데, - 빌드 프레임워크 없이 html, css, js만 사용하고 - DB 없는 스태틱 앱이니까 데이터베이스 api 호출 같은거 없이, 목업 형태로 만들어 줘야 해. - 모든 데이터는 사용자 PC 내에서만 처리되도록 만들어 주고. - 그리고 지금 윈도우 환경이니까, 완성 이후 서버 띄워서 테스트하려고 시도하지 말아. 내가 index.html 더블클릭해서 실행해 볼거야.
🧚‍♀️
왜 일반 프로그램 대신 '웹 앱'을 쓰는지 쉽게 설명해 드려요: AI 에이전트에게 지시할 때 '웹 앱'과 'HTML, CSS, JS'를 명시해주지 않으면 AI가 파이썬(Python)이나 C++ 같은 복잡한 언어를 쓰려 합니다. 이 경우 사용자가 로컬 기기에 수많은 파이썬 관련 패키지와 복잡한 구동 툴을 깔아야 하고 호환성이나 디버깅 문제에 부딪히며, 보안 경고로 파일 차단 등의 벽에 가로막힙니다. 웹 브라우저 기반 웹 앱은 더블클릭만으로 모든 컴퓨터와 스마트폰에서 안전하게 구동되므로 장벽이 없습니다.
🧚‍♀️
왜 빌드 프레임워크를 배제하는지 쉽게 설명해 드려요: 리액트(React), 뷰(Vue), 앵귤러(Angular) 등 현대 전문 개발에 활용되는 도구입니다. 전문적이고 대형 서비스를 만들 때에는 용이하지만, 일반 교육생들이 MVP 배포를 위해 사용하려 하면 컴퓨터에 개발 프로그램(Node.js 등)을 장시간 설치하는 무거운 전처리 단계가 필요하고, 추후 코드를 가볍게 열어서 글씨를 수정하는 것도 불가능해집니다. 순수 HTML, CSS, JS 파일은 메모장이나 일반 에디터로 수정하고 웹 새로고침만 하면 바로 동작하므로 비개발자가 고쳐 쓰기에 최상입니다.
🧚‍♀️
왜 서버와 DB가 필요 없는지 쉽게 설명해 드려요: 서버와 데이터베이스(DB)를 구축하면 관리비용(호스팅비)이 계속 청구되며, 정보 유출 보안 사고의 리스크가 생깁니다. 모든 데이터가 내 웹 브라우저 내의 로컬 임시 저장소(LocalStorage)에서만 굴러가도록 설계하면 서버비 0원에 완벽한 데이터 보안을 보장받을 수 있습니다.

1단계. GitHub CLI 로그인 연동

🧚‍♀️
이 단계가 필요한 이유를 쉽게 설명해 드려요: 내 컴퓨터의 명령 프롬프트(터미널)와 온라인 깃허브 웹 계정을 연동해, 내가 짠 코드를 마우스 클릭 없이 키보드 입력 한 번으로 깃허브 사이트에 안전하게 전송하는 인증 상태를 만듭니다.

단축키 Win + R ➡️ cmd 입력 후 엔터 ➡️ 검은 터미널 창에 아래 명령어를 복사해 입력하고 엔터를 누릅니다.

gh auth login

명령어 입력 후, 아래 상세 선택 옵션이 나오면 방향키와 엔터를 사용하여 차례로 골라줍니다:

  1. What account do you want to log into? ➡️ `GitHub.com` 선택 후 엔터
  2. What is your preferred protocol for Git operations? ➡️ `HTTPS` 선택 후 엔터
  3. Authenticate Git with your GitHub credentials? ➡️ `Y` 입력 후 엔터 (또는 `Yes` 선택)
  4. How would you like to authenticate GitHub CLI? ➡️ `Login with a web browser` 선택 후 엔터
  5. 화면에 8자리 인증코드(예: one-time code: XXXX-XXXX)가 출력되면 이를 따로 기억하거나 복사합니다.
  6. 키보드 엔터(Enter)를 한 번 더 누르면 웹 브라우저 로그인 창이 자동으로 열립니다.
  7. 브라우저 칸에 아까 띄워진 8자리 코드를 입력한 후 녹색 `Authorize GitHub` 버튼을 클릭하면 완료됩니다. 터미널 창에 ✓ Logged in as [본인아이디]가 뜨면 완료입니다.
⚠️ 1단계 빈출 에러 및 해결책 (클릭하여 열기)
* 오류: `'gh'은(는) 내부 또는 외부 명령... 이 아닙니다.`
  - 원인: GitHub CLI 프로그램이 아직 컴퓨터에 설치되지 않았거나, 설치를 방금 끝마쳤으나 환경 변수가 갱신되지 않은 구형 터미널 창을 쓰고 계신 상태입니다.
  - 해결책: cli.github.com에 접속해 설치를 마칩니다. 그 후 현재 켜져 있는 모든 터미널(CMD) 창을 완전히 닫고, 반드시 새 터미널 창을 새로 열어서 명령어를 입력하셔야 올바르게 실행됩니다.

* 질문: `인증코드(your device)는 어디로 오나요? 모바일 앱을 받아야 하나요?`
  - 해결책: 모바일 앱이나 폰 SMS를 뒤질 필요가 없습니다. 인증 코드는 명령어를 입력한 내 컴퓨터 터미널(검은 화면) 창에 이미 영문/숫자 조합(예: `XXXX-XXXX`)으로 출력되어 있습니다. 이 8자리 글자를 복사해 브라우저에 붙여넣기만 하면 인증이 통과됩니다.

2단계. GitHub 원격 저장소 생성

🧚‍♀️
이 단계가 필요한 이유를 쉽게 설명해 드려요: 내 웹 클라우드 공간(GitHub 사이트)에 내가 실습하여 만든 코드를 안전하게 보관할 '온라인 폴더(저장소)'를 새로 개설하는 단계입니다.

github.com/new 저장소 개설 링크로 들어갑니다. Repository name에 원하는 영문 이름(예: `fake_delivery`)을 쓰고, 스크롤을 내려 [Add a README file] 체크박스를 반드시 클릭하여 On(체크)으로 활성화합니다. 이후 최하단의 [Create repository] 버튼을 누릅니다.

⚠️ 2단계 빈출 에러 및 해결책 (클릭하여 열기)
* 오류: `README file을 생성 안 하고 개설하여 클론 시 깃 명령어 에러 발생`
  - 원인: 저장소를 개설할 때 README 파일을 추가하지 않으면 브랜치(main) 경로가 정상 생성되지 않아 빈 저장소로 복제가 실패하게 됩니다.
  - 해결책: 만약 README를 생성하지 않고 빈 저장소를 잘못 개설했다면 복잡한 깃 초기화 명령어를 쳐야 하므로 실패 확률이 높습니다. 이럴 때는 잘못 만든 저장소를 버리고, 반드시 `Add a README file`을 체크한 새 저장소를 새로 개설하는 것이 비전공자 입장에서 실패를 피하는 가장 빠른 지름길입니다.

3단계. 원격 저장소 클론 (Clone) 및 터미널 이동

🧚‍♀️
이 단계가 필요한 이유를 쉽게 설명해 드려요: 2단계에서 인터넷에 만든 온라인 폴더를 내 로컬 컴퓨터 바탕화면으로 다운로드(복제)하는 작업입니다. 이렇게 폴더를 동기화 복제해 와야 내 컴퓨터에서 수정한 코드들이 깃허브 사이트에 정상 연동되어 업로드됩니다.

방금 만든 깃허브 원격 저장소 웹페이지 우측 상단의 초록색 [<> Code] 버튼을 누르고 ➡️ [Local] ➡️ [GitHub CLI] 탭을 클릭한 뒤, 표시되는 복제 명령어(gh repo clone ...)를 복사합니다.

이후 아래의 단계에 따라 내 컴퓨터의 바탕화면에 저장소를 복제합니다 (Windows 11 기준):

  1. 바탕화면의 빈 공간을 마우스 우클릭한 뒤 ➡️ [터미널에서 열기]를 클릭합니다.
  2. 터미널 창에 방금 복사한 명령어(gh repo clone ...)를 마우스 우클릭이나 Ctrl + V로 붙여넣은 후 엔터(Enter)를 칩니다.
  3. 바탕화면에 저장소 이름으로 된 폴더가 생성되고 온라인의 파일들이 다운로드(복제)되면 성공입니다.
⚠️ 3단계 Windows 10 이하 구버전 경로 오류 및 해결 솔루션 (클릭하여 열기)
* 질문: `윈도우 10 등 구버전이라 바탕화면에 마우스 우클릭을 해도 '터미널에서 열기'가 뜨지 않습니다.`
  - 원인: 최신 윈도우 11 환경이 아닐 경우 마우스 우클릭으로 특정 폴더 위치의 터미널을 다이렉트로 실행할 수 없습니다.
  - 실제 해결 솔루션 (실패하지 않는 바탕화면 다운로드 방법):
    1. 윈도우 파일 탐색기를 열고 바탕화면으로 이동합니다.
    2. 바탕화면 빈 공간에서 마우스 우클릭 ➡️ [새로 만들기] ➡️ [폴더]를 클릭하여 '새 폴더'를 생성합니다.
    3. 생성한 새 폴더를 더블클릭하여 폴더 안으로 들어갑니다.
    4. 폴더 탐색기 맨 위쪽의 폴더 주소창(예: C:\Users\Username\Desktop\새 폴더)을 마우스로 클릭한 뒤 전체 주소를 복사합니다.
    5. 키보드 Win + R을 누르고 cmd를 입력해 명령 프롬프트(터미널) 검은 창을 켭니다.
    6. 터미널 창에 영어 소문자로 `cd `를 타이핑한 뒤 반드시 한 칸 띄우고(Space), 아까 복사한 주소를 마우스 우클릭으로 붙여넣고 엔터를 칩니다. (예: cd "C:\Users\Username\Desktop\새 폴더")
    (⚠️ 주의: 강사는 컴퓨터 권한 정책상 C드라이브 루트 경로(C:\ 바로 밑)에 실습 폴더를 만드는 것은 절대 금지하라고 강조했습니다. 윈도우 루트 경로는 보안 권한상 쓰기가 제약되므로 나중에 깃 전송이나 코드 수정 시 파일 쓰기 오류 버그가 대량 발생합니다.)
    7. 터미널 경로가 해당 폴더로 이동된 상태에서, 복사했던 `gh repo clone <유저명>/<저장소명> .` (저장소명 끝에 한 칸 띄고 마침표 .를 붙여서 실행하면 새 폴더가 이중으로 생성되지 않고 복제 파일만 해당 폴더에 쏙 다운로드됩니다)를 입력하고 엔터를 누릅니다.
    8. 이 단계가 성공적으로 끝나면 바탕화면에 만든 새 폴더 내부에 온라인 깃허브 저장소 파일들이 복제되어 다운로드되어 있습니다.

* 오류: `gh auth login을 마쳤는데도 gh repo clone 실행 시 'To get started with GitHub CLI, please run: gh auth login' 경고가 뜹니다.`
  - 원인: GitHub CLI 로그인 정보가 정상적으로 OS에 저장되지 않았거나 터미널 실행 세션에서 로그인 인증을 인식하지 못해 발생합니다.
  - 해결책: 현재 켜져 있는 명령 프롬프트(CMD) 창을 완전히 종료합니다. 새 명령 프롬프트를 띄운 후 `gh auth login`을 처음부터 한 번 더 차근차근 다시 진행해 주세요. 브라우저에서 승인을 완료하고 터미널에 `Logged in as`가 뜨는 것을 본 뒤에 클론을 실행하면 정상 동작합니다.

4단계. 프로젝트 폴더 및 터미널 열기 (Antigravity IDE 구동)

🧚‍♀️
이 단계가 필요한 이유를 쉽게 설명해 드려요: 내 로컬에 내려받은 프로젝트 코드를 AI 에디터(Antigravity)로 열고, 에이전트에게 시방서를 제공하여 코딩하고, 수정사항들을 다듬는 본격적인 앱 구현 단계입니다.

• 프로젝트 폴더 열기

  1. Antigravity IDE 프로그램을 실행합니다.
  2. 상단 탭 메뉴에서 `File (파일)` ➡️ `Open Folder (폴더 열기)`를 차례로 클릭합니다.
  3. 3단계에서 다운로드받은 바탕화면의 새 폴더를 선택하고 하단의 [폴더 선택]을 누릅니다.

• 에이전트 디렉팅 및 수정 다듬기 (실습용 가짜 배달 앱 구현)

  1. Antigravity 우측(또는 좌측) AI 에이전트 채팅창에 앞서 ChatGPT에서 뽑아낸 개발 시방서를 복사해 붙여넣습니다.
  2. 그 바로 아래에 아래의 웹 앱 코딩 프롬프트(전문)를 덧붙여 전송합니다.
    위 요구사항을 만족하는 웹 앱 만들어 주는데, - 빌드 프레임워크 없이 html, css, js만 사용하고 - DB 없는 스태틱 앱이니까 데이터베이스 api 호출 같은거 없이, 목업 형태로 만들어 줘야 해. - 모든 데이터는 사용자 PC 내에서만 처리되도록 만들어 주고. - 그리고 지금 윈도우 환경이니까, 완성 이후 서버 띄워서 테스트하려고 시도하지 말아. 내가 index.html 더블클릭해서 실행해 볼거야.
  3. 에이전트가 `index.html` 파일을 빌드할 때까지 대기합니다.
  4. 💡 앱 고도화: 최초 코드가 완성되면, 채팅창에 추가 프롬프트(예: "배달 현황 단계별 문구 수정해 줘", "주문 버튼 추가해 줘", "배경 테마를 깔끔하게 해 줘")를 계속 보내면서 앱 기능을 추가하고 요구에 맞춰 다듬는 작업을 진행합니다.
  5. 💡 결과물 공유: 코딩과 디자인이 마음에 들게 다듬어지면, 참가자들은 브라우저를 띄워 실행된 가짜 배달 웹앱 구동 화면을 캡처하여 디스코드 자유채팅방에 올려 서로 공유하고 피드백을 나누었습니다.

• IDE 내부 터미널 실행

  1. 깃허브 전송 명령을 입력하기 위해, Antigravity 상단 메뉴 탭에서 `Terminal (터미널)` ➡️ `New Terminal (새 터미널)`을 차례로 클릭합니다.
  2. 화면 하단에 검은색 명령 줄 입력 창(터미널)이 새로 활성화되면 명령어 입력 준비가 완료된 것입니다.

5단계. Git 커밋 및 깃허브 서버 업로드 (Push)

🧚‍♀️
이 단계가 필요한 이유를 쉽게 설명해 드려요: 내 로컬 컴퓨터 폴더에서 완성한 소스 코드를 온라인 깃허브 저장소(원격 클라우드 폴더)로 고스란히 복제해 전송하고 영구히 업로드(기록)하는 단계입니다.

Antigravity 하단 터미널(Terminal) 창에 아래 세 명령어를 입력하고 엔터를 칩니다.

🧚‍♀️
오류 없이 안전하게 명령어를 입력하는 팁을 드려요: 세 줄을 통째로 복사해서 한꺼번에 붙여넣어도 괜찮으나, 터미널 환경에 따라 줄바꿈 처리 오류나 명령 실행 지연 등 에러가 발생한다면 한 줄씩 개별적으로 복사하여 입력하시는 것이 가장 안전합니다.

1. 변경된 파일 전체를 깃 전송 대기열에 담아줍니다.

git add .

2. 변경 이력에 마킹을 남깁니다. (따옴표 내부는 커밋 메시지입니다.)

git commit -m "."

3. 대기 열의 소스 코드 파일들을 온라인 깃허브 원격 서버로 밀어 올립니다.

git push origin main
⚠️ 5단계 빈출 에러 및 해결책 (클릭하여 열기)
* 오류: `Author identity unknown (*** Please tell me who you are.)` 및 `unable to auto-detect email address`
  - 원인: Git을 처음 설치하고 실행할 때, 현재 사용자 이름과 이메일 정보가 Git 프로그램에 등록되어 있지 않아 보안상 업로드를 전면 거부하는 오류입니다.
  - 해결책: 터미널 창에 아래 두 명령어를 한 줄씩 차례로 입력하여 계정을 연동해 주세요. (주의: `< >` 꺾쇠 괄호는 제외하고 본인의 정보를 입력해야 합니다.)
git config --global user.name 본인의깃허브닉네임
git config --global user.email 본인의깃허브이메일
  (성공 시 아무 글씨도 뜨지 않고 다음 줄로 넘어갑니다. 아무 반응이 없는 것이 올바르게 등록된 상태입니다.)

* 경고: `warning: LF will be replaced by CRLF`
  - 해결책: 윈도우 환경과 타 OS 환경 간의 파일 줄바꿈 호환 처리를 위한 자동 변환 경고입니다. 정상적인 경고 문구이므로 안심하고 완전히 무시하셔도 됩니다.

* 오류: 세 줄 명령어를 한꺼번에 복사해서 넣었더니 먹통이 되는 경우
  - 해결책: 복사 시 한 번에 세 줄을 터미널에 밀어 넣으면 줄바꿈 문자 처리 지연 등으로 인해 오동작할 수 있습니다. 이 경우 한 줄씩 개별로 복사하여 터미널에 붙여넣고 엔터를 치며 순차적으로 입력하시면 오동작 없이 성공합니다.

6단계. GitHub Pages를 활용한 최종 웹 배포

🧚‍♀️
이 단계가 필요한 이유를 쉽게 설명해 드려요: 깃허브 원격 서버에 전송된 소스 코드를 서버(Pages) 기능을 구동시켜, 스마트폰이나 타인에게 카톡으로 보낼 수 있는 고유 공개 도메인 사이트 주소(URL)를 만드는 단계입니다.

본인의 GitHub 저장소 웹사이트 화면으로 들어가 ➡️ 상단 탭 [Settings] 클릭 ➡️ 좌측 메뉴 [Pages] 클릭 ➡️ Build and deployment 항목 내의 Branch 설정을 `None`에서 `main`으로 변경하고 우측의 [Save] 버튼을 누릅니다.

⚠️ 6단계 빈출 에러 및 해결책 (클릭하여 열기)
* 질문: `브랜치 main을 저장(Save)했는데도 사이트 주소가 바로 생기지 않습니다.`
  - 원인: 깃허브가 클라우드에서 배포 파일을 패키징하고 서버를 띄우는 데 약간의 연산 대기 시간(약 1~3분)이 필요합니다.
  - 해결책: 약 1~2분 정도의 여유를 두고 웹 브라우저 화면을 새로고침(F5)해 주시면 초록색 상자 안에 주소가 정상적으로 출력됩니다.

* 질문: `안티그래비티에서 코드가 수정되면 웹페이지에 자동으로 반영되나요?`
  - 해결책: 자동 반영되지 않습니다. 파일 수정 후에는 반드시 터미널에 업로드 명령어 세 줄(add, commit, push)을 한 줄씩 다시 입력해 주어야 원격 서버로 전송되어 사이트에 반영됩니다.

* 질문: `컴퓨터를 끄거나 브라우저 깃허브 창을 닫아도 제 사이트 링크는 열리나요?`
  - 해결책: 네, 그렇습니다. 내 컴퓨터가 아닌 깃허브의 클라우드 서버에서 상시 웹 서비스를 가동하는 형태이므로, 컴퓨터를 종료해도 전 세계 누구나 해당 URL로 언제든 정상 접속하여 웹앱을 사용할 수 있습니다.

Part 7. FAQ (기타 질의응답 요약)

실제 2주차 라이브 화상회의 세션 중 학습자들이 남겼던 애로사항과 이에 대한 대표님의 실시간 해결 답변 정리본입니다.

Q1. 작업(Working) 중에 에이전트에게 지시를 내리다가 상단의 AI 모델을 변경해도 동작하나요?

A. 네, 작업 도중에 모델을 변경하셔도 이전 상태가 손실되지 않으며, 변경 이후 새로 보내는 프롬프트 입력 시점부터 즉시 변경된 AI 모델이 적용되어 명령을 수행합니다.

Q2. 안티그래비티 IDE 화면이 VS Code와 디자인이 매우 흡사한 것 같습니다.

A. 네, Antigravity IDE는 널리 사용되는 오픈소스 코드 에디터인 VS Code(Visual Studio Code)를 포크(Fork)하여 제작된 프로젝트이므로 기본 레이아웃과 탭 구성이 유사합니다.

Q3. Gemini Canvas에서 다운로드한 URL을 다른 사람에게 공유하면 공유가 되나요?

A. 아니요. Gemini Canvas의 URL에는 개인 로그인 세션 정보가 포함되어 있어, 다른 사람이 접속할 경우 빈 화면이 뜨거나 접근이 거부됩니다. 화면 캡처 이미지를 보내거나 텍스트를 직접 복사하여 공유해 주셔야 합니다.

Q4. 코딩 에이전트 도구 중 Codex(코덱스) 같은 프로그램은 코딩 단계마다 사용자에게 물어보며 작업을 진행하던데, Antigravity는 왜 자율적으로 수정이 진행되나요?

A. 안티그래비티도 매 단계 승인을 거치게 설정할 수는 있으나, 비전공자 입장에서는 AI가 전문 기술 용어를 쓰면서 "이 작업을 수행해도 되냐"고 매번 물어보면 올바른 판단을 내리기도 어렵고 매 단계 승인 단추를 누르는 게 매우 번거롭습니다. 그래서 에이전트에게 "네가 판단해서 알아서 작성해라"라는 취지로 자유도를 높게 설정하여 운영하고 있습니다. 다만 이 경우 AI가 불필요하게 토큰을 태우거나 엉뚱한 코딩을 시도할 수 있으므로, 프롬프트 쪽에서 "백엔드 연산 금지", "프레임워크 사용 금지" 같은 명확한 제약 사항을 기재해 두어야 오동작을 억제할 수 있습니다.
