## 💡 Part 2. MVP와 PMF (Product Market Fit)

### 1. MVP (Minimal Viable Product, 최소 기능 제품)
* **개념**: 고객에게 가치를 제공할 수 있는 **최소한의 핵심 기능만 구현한 제품**입니다.
* **목적**: 완벽한 제품을 만들기 위해 시간과 비용을 쏟아붓기 전에, 시장에 빠르게 내놓아 고객의 진짜 반응을 확인하고 가설을 검증하는 도구입니다.
* **본질**: 화려한 디자인이나 복잡한 부가 기능은 걷어내고, 우리 비즈니스의 '핵심 가치'가 시장에서 작동하는지 검증하기 위한 최소한의 형태여야 합니다.

### 2. PMF (Product Market Fit, 제품-시장 적합성)
* **개념**: 우리의 제품이나 서비스가 **실제 시장의 강한 요구를 충족하는 상태**를 의미합니다.
* **PMF 달성 과정 (피드백 루프)**:
  PMF는 선형적인 단발성 과정이 아니라, 시장 반응에 따라 제품을 끊임없이 수정하고 최적화하는 **반복적인 순환 루프(Feedback Loop)**를 통해 달성됩니다.
  
  $$\text{[MVP 제작]} \rightarrow \text{[Release (출시)]} \rightarrow \text{[Test (검증)]} \rightarrow \text{[Feedback (피드백)]} \rightarrow \text{[Developed Product]} \rightarrow \text{(다시 Step 2(Release) 단계로 순환)}$$

* **PMF 루프의 핵심 구성 요소 (그로스 해킹과 피벗의 역할)**:
  PMF 피드백 루프가 실제로 작동하고 비즈니스를 성공으로 이끄는 핵심 개념들입니다. 이들은 별개의 개별 스토리가 아니라 PMF 달성을 위한 유기적인 전략입니다.
  * **그로스 해킹 (Growth Hacking - 루프의 가속화)**: 데이터 분석과 빠른 실험을 통해 이 피드백 루프를 극도로 빠르고 반복적으로 회전시키는 성장 전략입니다. 단순히 출시 후 덧붙이는 기법이라기보다는, 시장 반응 데이터를 토대로 핵심 기능을 지속적으로 보완하며 빠르게 PMF에 도달하도록 돕습니다.
  * **피벗 (Pivot, 사업 방향 전환 - 루프의 궤도 수정)**: 피드백 단계에서 가설이 시장에서 완전히 부정되었음을 깨달았을 때, 기존의 방향을 과감하게 전환하여 새로운 피드백 루프를 설계하는 의사결정입니다.
    * **피벗의 성공 사례 (YouTube의 기원)**:
      - **최초 기획**: 유튜브는 원래 "동영상 소개 기반의 데이팅 서비스"로 시작했습니다. ("여성이 직접 소개 영상을 올리면 대중적 인기를 얻을 것"이라는 가설)
      - **피드백과 실패**: 사용자들은 자신의 얼굴과 음성이 나오는 데이팅 소개 영상을 부담스러워하여 올리지 않았고, 처참한 피드백을 받았습니다.
      - **피벗(궤도 수정) 및 PMF 달성**: 이에 창업자들은 데이팅 컨셉을 완전히 버리고 **"주제 제한 없이 누구나 아무 동영상이나 자유롭게 공유하는 사이트"**로 피벗을 단행했습니다. 시장 피드백을 루프에 신속히 수용하여 제품 방향성을 전면 수정한 결과, 비로소 시장의 진짜 수요와 만나(PMF 달성) 오늘날 세계 1위 동영상 플랫폼이 되었습니다.

## 🪙 Part 3. 시장조사를 위한 8가지 질문과 시장 규모 측정

### 1. 기획 단계에서 스스로 던져야 할 8가지 질문
> 💡 답변을 작성하다가 **단 하나라도 명확히 막힌다면 처음부터 기획을 다시 검토**해야 합니다.
1. **누구를 위한 서비스인가?** (명확한 타겟 고객 설정)
2. **고객이 유의미한 규모의 집단인가?** (시장성의 크기 검증 - 💡 **Gemini Canvas 활용 권장**)
3. **그 사람들의 어떤 문제를 해결해 주는가?** (문제의 명확성)
4. **정말로 그게 문제가 맞나?** (고객 관점의 진짜 고통 검증)
5. **문제를 어떻게 해결할 것인가?** (해결책의 실효성)
6. **그 해결 수단이 어떤 특별한 가치를 갖나?** (차별성 및 독점적 가치)
7. **해결 수단을 어떻게 고객에게 전달할 것인가?** (마케팅 및 유통 채널)
8. **경쟁자 대비 어떤 장점이 있나?** (경쟁 우위 분석)

### 2. 시장 규모의 3단계 레이어 (SOM, SAM, TAM)
* **TAM (Total Available Market, 전체 시장 규모)**: 우리 제품이 속한 전체 산업 영역입니다. (예: 전체 자동차 산업)
* **SAM (Serviceable Available Market, 도달 가능한 시장 크기)**: 우리 비즈니스가 이론적으로 도달 가능한 영역입니다. (예: 글로벌 및 국내 전기차 시장)
* **SOM (Serviceable Obtainable Market, 즉각 점유 가능한 시장)**: 우리의 자원으로 즉시 획득 가능한 실제 시장입니다. (예: 국내 전기 SUV 시장 중 4천만 원 이하의 40대 타겟)
* **Niche Market (틈새시장)**: 대기업이 쉽게 진입하지 못하는 좁은 시장 영역입니다.
* ⚠️ **MVP 제작의 원칙**: MVP를 기획하고 만들 때에는 거창한 TAM이나 SAM 규모를 논하기보다, **틈새시장(Niche Market)과 즉각 점유가 가능한 최대 SOM 범위까지만 한계선으로 설정**하고 검증해야 합니다.

#### 1단계. GitHub CLI 로그인 연동

<div style="border-left: 4px solid #C084FC; padding: 8px 16px; margin: 16px 0; display: flex; gap: 12px; align-items: flex-start;">
    <div style="font-size: 1.5rem; line-height: 1; flex-shrink: 0;">🧚‍♀️</div>
    <div style="flex-grow: 1; font-size: 0.95rem; color: #581C87; line-height: 1.6; word-break: keep-all;">
        <strong>요정의 넛지 (이 단계가 필요한 이유를 쉽게 설명해 드려요)</strong>: 내 로컬 컴퓨터의 터미널 창과 웹사이트 GitHub 계정을 연결하여, 내가 작성한 코드를 번거로운 마우스 클릭 없이 원클릭 명령어로 깃허브 사이트에 전송(업로드)할 수 있도록 보안 연동을 설정하는 단계입니다.
    </div>
</div>
* 윈도우 단축키 `Win + R` -> `cmd` 입력 후 엔터 (명령 프롬프트 실행)
* 터미널에 아래 명령어 입력 후 엔터:
  ```bash
  gh auth login
  ```
* **이후 진행되는 상세 연동 옵션 선택 순서**:
  1. `What account do you want to log into?` -> 방향키로 **`GitHub.com`** 선택 후 엔터.
  2. `What is your preferred protocol for Git operations?` -> 방향키로 **`HTTPS`** 선택 후 엔터.
  3. `Authenticate Git with your GitHub credentials?` -> **`Y`** 입력 후 엔터 (또는 `Yes` 선택).
  4. `How would you like to authenticate GitHub CLI?` -> 방향키로 **`Login with a web browser`** 선택 후 엔터.
  5. 터미널 창에 8자리 일회용 인증코드(예: `one-time code: XXXX-XXXX`)가 표시됩니다.
  6. 키보드 엔터(Enter)를 한 번 더 누르면 웹 브라우저 로그인 창이 자동으로 열립니다.
  7. 브라우저 화면에 터미널에서 보았던 8자리 인증코드를 입력하고 녹색 **`Authorize GitHub`** 버튼을 클릭합니다.
  8. 터미널 창으로 돌아왔을 때 `✓ Logged in as <계정명>`이 표시되면 연동 완료입니다.
  
<details>
<summary><strong>⚠️ 1단계 주요 트러블슈팅 및 빈출 에러 (클릭하여 열기)</strong></summary>
<div style="padding-left: 10px; margin-top: 10px;">

* **오류: `'gh'은(는) 내부 또는 외부 명령, 실행할 수 있는 프로그램, 또는 배치 파일이 아닙니다.`**
  - **원인**: GitHub CLI 프로그램이 아직 컴퓨터에 설치되지 않았거나, 방금 설치를 완료했으나 켜져 있는 터미널(CMD) 창에 환경 변수가 갱신되어 반영되지 않은 상태입니다.
  - **해결책**: [cli.github.com](https://cli.github.com/)에서 전용 설치 파일을 다운로드하여 설치를 진행합니다. 설치 완료 후 **현재 열려 있는 터미널 창을 완전히 종료하고, 반드시 새 터미널 창을 열어** 명령어를 입력해 주세요.
* **질문: `인증코드(your device)는 어디로 전송되나요? 모바일 인증 앱을 받아야 하나요?`**
  - **원인**: 브라우저 연동 창에서 "your device로 코드를 발송했다"고 하여 스마트폰 SMS나 메일을 찾는 혼동이 발생합니다.
  - **해결책**: 스마트폰을 볼 필요가 없습니다. 로그인 인증코드 8자리는 **명령어를 실행한 내 컴퓨터 터미널(검은 창) 화면에 이미 영문/숫자 조합(예: `XXXX-XXXX`)으로 출력**되어 있습니다. 이 코드를 복사하여 웹 브라우저의 GitHub 권한 승인 페이지에 그대로 입력하고 `Authorize GitHub` 버튼을 누르면 끝납니다. 모바일 앱을 설치할 필요가 없습니다.

</div>
</details>

<details>
<summary><strong>⚠️ 3단계 주요 트러블슈팅 및 빈출 에러 (클릭하여 열기)</strong></summary>
<div style="padding-left: 10px; margin-top: 10px;">

* **질문 (Windows 10 등 구버전 사용자): `바탕화면 마우스 우클릭에 '터미널에서 열기'가 없어요. 경로 이동을 어떻게 하나요?`**
  - **원인**: 최신 윈도우 11이 아닌 구버전 OS 환경에서는 바탕화면이나 특정 폴더에서 바로 터미널을 실행하는 기본 옵션이 없습니다.
  - **실제 해결 솔루션 (실패하지 않는 바탕화면 다운로드 방법)**:
    1. 윈도우 파일 탐색기를 열어 **바탕화면**으로 이동합니다.
    2. 바탕화면 빈 공간에서 마우스 우클릭 -> **[새로 만들기] -> [폴더]**를 클릭하여 '새 폴더'를 생성합니다.
    3. 생성된 새 폴더를 더블클릭하여 안으로 진입합니다.
    4. 폴더 탐색기 맨 위쪽의 **폴더 주소창(예: C:\\\Users\\Desktop\\새 폴더)**을 마우스로 클릭하여 주소 텍스트를 전체 복사합니다.
    5. 키보드 `Win + R`을 누르고 `cmd`를 입력한 뒤 엔터를 눌러 검은색 터미널 창을 실행합니다.
    6. 터미널 창에 영어로 **`cd`를 적고 반드시 한 칸 띄운(Space) 다음, 마우스 우클릭이나 단축키로 복사한 주소를 붙여넣고 엔터**를 누릅니다. (예: `cd "C:\\\Users\\Name\\Desktop\\새 폴더"`)
       *(⚠️ **주의**: 강사는 컴퓨터 권한 정책상 C드라이브 루트 경로(`C:\\\` 바로 아래)에 실습 폴더를 만들고 코딩하는 것은 절대 권장하지 않는다고 했습니다. 루트 경로는 윈도우 보안 정책상 파일 수정에 제한을 받기 때문에, 나중에 깃 명령어로 업로드할 때 수정 권한 부족 에러 등 예기치 못한 버그가 대량 발생할 수 있습니다.)*
    7. 터미널 경로가 해당 폴더로 이동된 상태에서, 아까 복사해 두었던 **`gh repo clone <유저명>/<저장소명> .`** (저장소명 끝에 <strong>한 칸 띄고 마침표 <code>.</code></strong>를 붙여서 실행하면 해당 폴더 자체에 다운로드됩니다)를 입력하고 엔터를 누릅니다.
    8. 이 단계가 성공적으로 끝나면 바탕화면에 만든 새 폴더 내부에 온라인 깃허브 저장소 파일들이 복제되어 다운로드됩니다.
* **오류: `gh auth login을 마쳤는데도 gh repo clone 실행 시 'To get started with GitHub CLI, please run: gh auth login'이라는 로그아웃 경고가 뜹니다.`**
  - **원인**: GitHub CLI 로그인 정보가 정상적으로 OS에 저장되지 않았거나 터미널 실행 세션에서 로그인 인증을 인식하지 못해 발생합니다.
  - **해결책**: 현재 켜져 있는 명령 프롬프트(CMD) 창을 완전히 종료합니다. 새 명령 프롬프트를 띄운 후 `gh auth login`을 처음부터 한 번 더 차근차근 다시 진행해 주세요. 브라우저에서 승인을 완료하고 터미널에 `Logged in as`가 뜨는 것을 본 뒤에 클론을 실행하면 정상 동작합니다.

</details>

<summary><strong>⚠️ 5단계 주요 트러블슈팅 및 빈출 에러 (클릭하여 열기)</strong></summary>

## ❓ Part 7. FAQ (기타 질의응답 요약)

*(💡 GitHub Pages 배포 및 실습 과정에서 발생한 각종 CLI/Git 오류, 환경 변수 에러, Windows 10의 특수 해결 솔루션 등은 참가자들의 복습 편의성을 극대화하기 위해 **Part 5 실습 가이드 내부의 각 단계별 [CAUTION] 영역에 100% 밀착 병합 정리**되었습니다. 하단 FAQ는 그 외의 기타 운영 및 이론 관련 보충 질의응답입니다.)*

* **Q1. Working 중에 에이전트에게 지시를 내리다가 상단의 AI 모델을 변경해도 동작하나요?**
  * **A.** 네, 작업 도중에 모델을 변경하셔도 이전 상태가 손실되지 않으며, 변경 이후 새로 보내는 프롬프트부터 변경된 AI 모델이 적용되어 명령을 수행합니다.
* **Q2. 안티그래비티 IDE 화면이 VS Code와 디자인이 매우 흡사한 것 같습니다.**
  * **A.** 네, Antigravity IDE는 널리 사용되는 오픈소스 코드 에디터인 VS Code(Visual Studio Code)를 포크(Fork)하여 제작된 프로젝트이므로 기본 레이아웃이 유사합니다.
* **Q3. Gemini Canvas에서 다운로드한 URL을 다른 사람에게 공유하면 공유가 되나요?**
  * **A.** 아니요. Gemini Canvas의 URL에는 개인 로그인 세션 정보가 포함되어 있어, 다른 사람이 접속할 경우 빈 화면이 뜨거나 접근이 거부됩니다. 화면 캡처 이미지를 보내거나 텍스트를 직접 복사하여 공유해 주셔야 합니다.
* **Q4. 코딩 에이전트 도구 중 Codex(코덱스) 같은 프로그램은 코딩 단계마다 사용자에게 물어보며 작업을 진행하던데, Antigravity는 왜 자율적으로 수정이 진행되나요?**
  * **A.** 안티그래비티도 매 단계 승인을 거치게 설정할 수는 있으나, 비전공자 입장에서는 AI가 전문 기술 용어를 쓰면서 "이 작업을 수행해도 되냐"고 매번 물어보면 판단하기도 어렵고 매우 번거롭습니다. 그래서 에이전트에게 "네가 판단해서 알아서 작성해라"라는 취지로 자유도를 높게 설정하여 운영하고 있습니다. 다만 이 경우 AI가 불필요하게 토큰을 태우거나 엉뚱한 코딩을 시도할 수 있으므로, 프롬프트 쪽에서 "백엔드 연산 금지", "프레임워크 사용 금지" 같은 명확한 제약 사항을 기재해 두어야 오동작을 억제할 수 있습니다.
